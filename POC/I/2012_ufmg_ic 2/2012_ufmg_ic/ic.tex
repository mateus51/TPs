\bf 
{\Large Melhorias e Implementação de Novas Características para}  \\[.3cm] {\Large BugMaps: Uma Ferramenta para Visualização e Análise de Bugs} \\[.5cm]

{\large Marco Túlio Valente - mtov@dcc.ufmg.br}\\[.5cm]

\end{center}

\noindent{\bf Caracterização do Problema:} Atualmente existe um grande número de ferramentas de análise de software~\cite{nierstrasz2005, wettel2009, sonar2011}. Tais ferramentas usam diferentes tipos de informação sobre a estrutura e o histórico de um sistema. Basicamente, estas ferramentas são usadas para analisar a evolução de um software, gerenciar a qualidade do código fonte, calcular métricas, analisar regras de codificação, etc. Em geral, estas ferramentas ajudam engenheiros de software a entender grandes quantidades de dados vindas de repositórios de software. 

Por outro lado, um fonte de informação que tem sido pouco explorada pelas ferramentas de análise existentes são os {\em bugs} de um sistema. Algumas ferramentas já analisam esta informação, mas ainda pouco se sabe sobre o comportamento evolucionário, tempo de vida, distribuição e estabilidade dos {\em bugs}~\cite{dambros2010, dambros2007}. Além disso, raciocinar sobre {\em bugs} é uma tarefa inerentemente complexa, porque {\em bugs} são intangíveis, invisíveis e difíceis de serem rastreados. Particularmente, tal análise é complexa pois envolve: (i) recuperação de dados do sistemas de gerenciamento de bugs e sistemas de controle de versão; (ii) mapeamento de {\em bugs} para defeitos em módulos de software; (iii) processamento de dados para extrair e raciocinar sobre informações relevantes.\\

\noindent{\bf Pesquisa Desenvolvida:} A primeira versão da ferramenta chamada de BugMaps já se encontra implementada e publicada~\cite{hora12}. A primeira versão foi desenvolvida em parceria com o INRIA França, projeto "Pequi", Cooperação FAPEMIG -- INRIA, Edital 22/2010, proc APQ-00214-11. Até o presente momento BugMaps possui as seguinte características:
\begin{itemize}
\item A ferramenta automaticamente extrai uma série temporal com o número de defeitos a partir de informações disponíveis em sistemas de gerenciamento de {\em bugs} (Jira e Bugzilla) e sistemas de controle de versão (CVS e SVN). 
\item A ferramenta integra modelos extraídos do código fonte com a série temporal número de defeitos.
\item A partir da integração, a ferramenta fornece um conjunto de visualizações que ajudam desenvolvedores de software e gerentes na resposta das seguintes questões: (a) Quais são os módulos envolvidos na correção de {\em bugs}? (b) Qual é o tempo de vida de um {\em bug}. (c) Qual é o período que um módulo apresentou mais {\em bugs}. (d) Quais módulos são estáveis ou instáveis com relação aos {\em bugs}? (e) Qual é o número total de {\em bugs} de um módulo? \\ 
\end{itemize}



\noindent{\bf Próximos Passos:} Com o objetivo de tornar a ferramenta mais profissional e consequentemente usual, pretende-se implementar as seguintes atividades:
\begin{itemize}
\item Automatizar a extração dos dados vindos de sistemas de gerenciamento de {\em bugs} e sistemas de controle de versão. Atualmente este processo é manual.
\item Melhorar o processo de integração de modelos extraídos do código fonte com a série temporal número de defeitos. Atualmente este processo está sendo feito por meio de três arquivos .csvs contendo informações sobre os defeitos, a idéia é fazer a integração por meio de um único arquivo XML.
\item Melhorar as visualizações existentes acrescentando informações sobre o teste de Granger. O teste de Granger foi objeto de estudo do trabalho Couto {\em et al.}~\cite{couto12}. Basicamente este trabalho usa Granger para encontrar relações causais entre métricas de código fonte é {\em bugs}. A idéia é instruir a ferramenta BugMaps com informações sobre quais propriedades do código fonte, de acordo com o Granger, foram as causas dos {\em bugs}. 
\item Criar uma documentação contendo os procedimentos para instalação, configuração e uso. 
\item Criar um site para disponibilização da ferramenta contendo a documentação do item anterior.

\end{itemize}
